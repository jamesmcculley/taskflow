# Decisions

- **2026-07-20** — Closed the remaining half of "edit a done date via markdown": v0.13.0's `findUnloggedCompletions` only caught completions with *no* log entry yet. A new `findStampDrift` handles the other case — an *already-logged* completion whose ✅ stamp gets hand-edited afterward — by comparing the log entry's day against the line's current stamp and, on mismatch, calling the same `editCompletionDate` the "Edit date…" UI uses (so the daily journal moves too, not just the log). Deliberately skipped when a task has more than one 'done' entry (recurring tasks accumulate one per past occurrence) since there's no way to know which historical entry a single current stamp corresponds to — that disambiguated case is what the per-entry "Edit date…" action is for.

- **2026-07-20** — Fixed a real bug (found by auditing a live vault): only the plugin's own actions ever wrote to the completion log, so a task checked off via Obsidian's *native* checkbox (or hand-typed `[x]`, or a change synced in from another device) never got a ✅ stamp or a History entry — it just silently vanished into "done, but never logged." The indexer now runs a `findUnloggedCompletions` pass on every file it indexes: any done/cancelled task with no matching log entry gets one, using its own ✅ stamp date if present (so pre-existing old completions keep their real date) or today's date as a last resort (batched into the same kind of write `assignBlockIds` already does for missing block IDs). This also runs during the first full scan, so pre-existing `[x]` tasks in a vault picking up the plugin for the first time get backfilled into History too — deliberately consistent with how missing block IDs already get silently backfilled on first scan, not something to gate behind a setting. Recurring tasks are *not* auto-advanced by this path (only the plugin's own `completeTask` knows how); a natively-checked recurring task is logged and stamped like any other done task, but its line and dates are left exactly as found. `recordCompletion`'s parameter loosened from `Task` to `Pick<Task, 'id'|'title'|'project'>` so both the plugin's own actions and this passive detection path can share it via a new public `recordExternalCompletion` wrapper.

- **2026-07-20** — History entries can have their completion date corrected via **Edit date…** (right-click), identified the same way `removeLogEntry` already does (taskId + that entry's completedAt — not just taskId, since a recurring task's ID repeats across many historical log entries). If the entry is still the task's *live* completion (line still carries that ✅ stamp), the stamp is rewritten too; if it's a historical entry whose line has since moved on (recurrence rewrote it to the next occurrence), only the log and daily-note journal move to the corrected day — there's no live stamp to touch. `DailySync.record` was refactored to take raw fields (taskId/title/projectPath) instead of a live `Task`, since correcting a historical entry has no live Task to hand it.

- **2026-07-20** — Every task row now shows a source-note chip (`#Note Name`, derived from `task.file`, not written to markdown) so tasks in mixed-origin views (Inbox, Upcoming, Whenever…) show where they live without opening them. Replaces the old project-only chip; project note tasks show the same chip since `project.name` already equals the file's basename. Hidden (`hideSource`) inside a note's own project/area view, where it would just repeat the page you're on. Styled tag-like (accent color, `#` prefix) but kept visually distinct (italic) from real `#tags` so it's never mistaken for something you typed or something clickable-as-a-filter.
- **2026-07-20** — Inbox redefined as a triage holding area, not "wherever a non-project task happens to live": `selectInboxTasks` now also excludes tasks with a scheduled or due date. Scheduling a task moves it out of Inbox (into Today/Upcoming) without moving the file; clearing the date moves it back automatically, so a task can never get silently lost by half-editing it. This only changes the selector — no markdown or file-location logic changed — and the review-mode Overdue step no longer double-lists an unfiled overdue task under Inbox too, since it now belongs only to Overdue.

- **2026-07-20** — `taskflow: false` frontmatter now also accepts the quoted string form (`taskflow: "false"`, any case, plus `"ignore"`/`"off"`/`"no"`). Obsidian's property UI writes a quoted string when the field is typed as a "Text" property, which YAML parses as a string rather than a boolean — the original check only matched the boolean, so a note excluded through the property UI silently stayed indexed. Found via a real vault note ("Test Note.md") where this happened.

- **2026-07-20** — Checkbox exclusion at three scopes: `#notask` tag per line, `taskflow: false|ignore|off` frontmatter per note, and an excluded-folders setting (prefix match on path segments; changing it triggers a full rescan that also evicts newly excluded files from the index). Exclusion happens before ID assignment, so excluded checkboxes never get `^t-` refs appended.

- **2026-07-19** — M7: priority is a standalone `!!`/`!!!` token (a trailing "!" in a title never counts); priority sorts above manual order, and a scheduled time (optional `HH:mm` after ⏳'s date) sorts within the day — compared by codepoint, not localeCompare, whose collation broke the untimed-last sentinel. Capture's recurrence extraction bounds the phrase with a vocabulary whitelist before rrule validates it, because fromText silently ignores trailing junk. Backdated completions timestamp at noon local on the chosen day. Board view is wide-layout only; dragging between columns rewrites the task under the target heading (the no-heading column inserts above the first heading).

- **2026-07-19** — M6: task-level Someday is a plain `#someday` tag (typeable anywhere, incl. mobile) rather than a new emoji token; it's hidden from tag chips and unions with someday-project membership. Tonight is a `🌙` flag; enabling it also schedules today when the task isn't already due/scheduled today. Checklist items are checkboxes nested under a checkbox (through any bullet levels, attached to the root task); they still get block IDs but are no longer independent tasks — they don't appear in lists or project progress. After-completion repeats use the `after done` / `after completion` suffix on the rrule text. Selection moved from React context into the zustand store so Quick search can navigate-and-select.

- **2026-07-19** — Daily-note journal lines are plain `- ✅ HH:mm …` list items, not checkboxes, so the indexer never re-ingests them as tasks; a hidden `%%taskId%%` comment makes removal on uncomplete exact. Recurring tasks completed twice a day produce two lines; removal takes the last. Daily sync is on by default (the feature was requested as a default workflow) with a settings toggle.
- **2026-07-19** — Pinned filters match open tasks only and AND their criteria; tags use prefix matching for nested tags (`dev` matches `dev/backend`). Filters live in data.json (index-owned UI state, like sort order) — not in markdown.

- **2026-07-19** — M4: single click now selects a task; opening the note moved to double-click/Enter. Manual drag-reorder writes list-scoped integer order values to the index only — markdown line order is untouched, so reordering never causes file churn. Reordering is enabled in manually-ordered lists (Inbox, Today, Whenever, Someday, project sections) and not in date-driven Upcoming. Completion linger is 1.6s.

- **2026-07-19** — The indexer reconciles the History with markdown on every (re)index: completions undone outside the plugin (native checkbox in the note, manual edit) are pruned, and closed non-recurring tasks keep only their latest entry. Recurring tasks are exempt (log is their only history). Orphaned entries for deleted lines are kept as history but removable via right-click → Remove from History.

- **2026-07-18** — `every Nth <weekday>` recurrence shorthand is normalized to rrule's `every month on the Nth <weekday>` (monthly semantics, the common expectation); rrule's own parser rejects the short form. Unparseable recurrence text falls back to a plain completion with a Notice rather than blocking the checkbox. Recurrence text must start with `every` — rrule's fromText silently returns a yearly rule for arbitrary junk.
- **2026-07-18** — Next occurrence = first strictly after max(anchor, today), with dtstart at the anchor: overdue repeats skip into the future but stay aligned to the original pattern, and completing early still advances past a future anchor.
- **2026-07-18** — The History reads only the index completion log (per spec); tasks checked off by hand in markdown don't appear there. Uncompleting removes the task's most recent log entry. Recurring completions exist only in the log, since the line is rewritten as the next occurrence.
- **2026-07-18** — Whenever includes tasks that have a due date but no scheduled date (spec says "no scheduled date"; a deadline alone doesn't make a task "scheduled").
- **2026-07-18** — Upcoming/Today placement uses the earlier of scheduled/due (same effective-date rule the M2 tabs used). Upcoming labels: Tomorrow, weekday names through the end of the current Monday-start week, then `Jul 25` (with year if different).
- **2026-07-18** — Narrow/wide layout switches at 480px container width via ResizeObserver: narrow = nav behind a menu toggle, wide = two-pane.

- **2026-07-18** — Plan adjusted: the panel should provide a complete, polished sidebar UI (see ROADMAP.md). Consequences: the M2 This Week/Future tabs are removed in M3 (Upcoming covers both, date-grouped); Areas join the sidebar via `area: <name>` project frontmatter; a new M4 milestone owns visual fidelity and interactions (animations, magic plus, keyboard nav, drag reorder).

- **2026-07-18** — Added "This Week" and "Future" tabs (user request, ahead of M3's Upcoming). Definitions: placement date = the earlier of scheduled/due; This Week = dated after today through Sunday (Monday-start weeks); Future = dated beyond that. Today is excluded (it has its own tab) and undated tasks are excluded (they become Whenever/Someday in M3). M3's date-grouped Upcoming still lands as specced.
- **2026-07-18** — Mutations live in a `TaskActions` module on the plugin rather than inside the zustand store: the store stays a pure data container (unit-testable without Obsidian), while actions need vault access. Views still funnel every mutation through this one module.
- **2026-07-18** — `moveToProject` inserts into the destination before deleting from the source; a crash in between leaves a duplicate rather than a lost task. Moved tasks are normalized to top-level items, and nested child items do not travel with a moved parent (revisit if checklists become first-class).
- **2026-07-18** — Cancelling writes no ✅ stamp to the line (cancellations aren't dated); the timestamp is recorded index-only so the History can group cancelled items by day.
- **2026-07-18** — Editor commands ("… task under cursor") are the "sensible default" targets for hotkey-bindable mutations; sidebar tasks use the context menu and checkbox instead.
- **2026-07-18** — Capture parse precedence: `!due <date>` first, then `>Project`, then `#tags`, then the first remaining natural-language date (chrono, forward-biased) becomes the scheduled date. Project names resolve exact → prefix → substring, falling back to Inbox with a note in the preview.

- **2026-07-18** — Unknown checkbox statuses (e.g. `- [/]`, `- [>]` from other plugins/themes) are indexed as `todo` rather than skipped, so tasks never silently disappear from views. Only `x`/`X` map to done and `-` to cancelled.
- **2026-07-18** — If a task line already carries any block reference (`^whatever`), it is reused as the stable ID instead of appending a second `^t-xxxxxx` — a markdown line can only have one block ref, and existing refs are already stable.
- **2026-07-18** — Tasks in ordinary notes (no `type: project` frontmatter, not `Inbox.md`, not a daily note) are treated as Inbox items; the spec only distinguishes project notes from everything else, and `project: undefined` already means Inbox in the data model.
- **2026-07-18** — Recurrence text (`🔁 …`) is captured up to the next metadata emoji, `#tag`, or trailing block ref, since rrule text never contains those tokens.
- **2026-07-18** — Tokenizer and store selectors are pure modules with zero `obsidian` imports so unit tests need no API mocking; only the indexer/view layers touch the Obsidian API.
- **2026-07-18** — Daily notes are detected by `YYYY-MM-DD` basenames (the Obsidian default); the Daily Notes core-plugin format setting can be wired in later if needed.
